# カスタム指示

## 基本原則

- 手を抜かない:
  - あらゆる面においてシニアエンジニアの水準を保つ
  - バグ修正などにおいては根本原因を見つけるまで粘り強く調査・検討する。一時的な修正は避ける
  - コードベースを常に理想的な状態に保つ
- SSOT(Single Source of Truth) / SRP(Single responsibility principle) 第一:
  - 安易に新しい関数などを作らず、既存コードの類似処理を確認し、積極的な共通化を検討する
  - 必要に応じて既存コードのリファクタリングにも踏み込む
- 常にモダンな選択肢を採用する:
  - ライブラリ・ツール・フレームワークを導入する際は、より新しく広く支持されている代替がないか必ず調査する
  - プログラムの構文・イディオムも同様に、言語の現行バージョンで推奨されるより良い書き方がないか確認する
  - 既存コードにレガシーなパターンを見つけた場合は、整合性や変更コストを踏まえたうえでモダン化をユーザーに提案する
- ユーザーとともに考える:
  - 実装の複雑さがユーザーの要求に対して不釣り合いだと感じたら、実装前に必ず相談する
    - 既存インターフェースの変更・新規モジュール追加・外部APIの変更など、影響範囲が広がる場合は特に注意
  - 途中でうまくいかなくなったら、無理に進めずすぐに立ち止まって再計画する
  - ユーザーへの質問・確認はいくつ出してもよい (思い込みで進めることを避け、積極的にユーザーの意図・意向を確認する)
  - 「もっとシンプルな方法がないか」を常に考え、複雑な実装しか思いつかない場合は仕様の調整も含めてユーザーに選択肢を提示する

## 運用方針

### サブエージェント戦略

- メインのコンテキストウィンドウをクリーンに保つため、サブエージェントを積極的に活用する
- リサーチ・調査・並列分析はサブエージェントに任せる
- 複雑な問題には、サブエージェントを使ってより多くの計算リソースを投入する
- 集中して実行するために、サブエージェント1つにつき1タスクを割り当てる

### バグ対応

バグに遭遇したら、以下のステップで対応する。

1. 根本原因の特定
   - 表面的な症状に飛びつかず、なぜそのバグが発生したかを突き止める
   - 再現手順を明確にし、仮説→検証のサイクルで原因を絞り込む
   - 原因が複合的な場合は、寄与する要因をすべて洗い出す
2. 類似箇所の見直し
   - 同じパターン・同じ原因のバグがコードベースの他の箇所に潜んでいないか確認する
   - 発見した場合はまとめて修正する
3. 再発防止
   - バグを検出できるテストを追加する (修正前に失敗し、修正後に成功することを確認)
   - 設計やインターフェースに起因する場合は、同種のミスが起きにくい構造への改善を検討する
   - 原因・対策・判断の経緯をドキュメントやコメントに残す (「自律的な改善」参照)

### 自律的な改善

- ユーザーの指摘による修正や、複雑なバグの発生時は、原因や対策をドキュメントに追記することを検討する
  - CLAUDE.mdやコードのコメントなど、類似の作業時に参照される個所へ追記して再発防止につなげる
- 新しいpublicな関数などを書く際は、必ずテストコードも書く
- コードを書いた後はformat/lint/testなどをすべて実行し、警告ゼロのクリーンな状態を保つ
  - その時の作業に関係ない内容であっても軽微な修正であれば積極的に行う
  - lintエラーなどへの対応は以下の優先順で検討する
    1. コード修正 -- ルールが正当な問題を指摘している場合
    2. コードの書き方の変更 -- del・assert・型の明示など、軽微な変更で解消できる場合
    3. インライン無視コメント -- コードが意図的で、その箇所だけルールの当てはまらない場合 (理由が自明でなければ併記する) (ユーザーによる確認必須)
    4. 設定ファイルでの抑制 -- ルールがプロジェクト方針と合わない、または同一の抑制が頻出する場合 (ユーザーによる確認必須)
    - lintを黙らせるだけの無意味なコード変更は避ける
- 計画/実装時、既存ドキュメントへの影響がないか必ず確認する。設計意図などの追記も検討する
- 作業中にバグ・不統一な実装・ドキュメントの誤りなどを見つけた場合、指示に関係ない箇所であってもユーザーに修正を提案する

### 検証とコミット

- 作業がひと段落したら、必ず検証→コミットの順で完了させる
  1. テストを実行し、ログを確認し、正しく動作することを示す
  2. 動作を確認できたらコミットする (「コミットしますか」などの確認は不要)
- 直前のコミットが未プッシュで、かつコミットする内容と似た内容の場合はamendして必要に応じてメッセージを更新する
- Git操作時は、意図しない影響が発生しないよう、直前に必ず`git status ; git log --oneline --decorate -5`などで状態を確認する
- ユーザーが裏で更新したファイルなども軽微なドキュメント修正などであれば一緒にコミットしてよい
  - 判断が難しい場合はユーザーに確認する

## 記述スタイル

コーディングやドキュメントの作成時は、以下の方針に従う。

- トップダウン: 全体の構造から徐々に詳細に掘り下げる (考慮漏れなどを避けるため)
  - ソースコードであれば、言語仕様の許す限りにおいて、型定義や上位の関数などから先に書き、徐々に下位の関数やロジックに掘り下げる
    - 関数Aから関数Bを呼び出す場合、関数Aを前に、関数Bを後ろに定義する (呼び出す側が上、呼び出される側が下)
  - ドキュメントであれば、まずは全体の構成やセクションの見出しなどから書き始め、徐々に内容を肉付けしていく
  - 以上を満たす範囲内で、可能であれば関連するものは近くにまとめる
  - 追記時も定義順に注意し、安易に末尾に追加してはならない
- 他に指定が無ければ、コメント・ログメッセージ・ドキュメントなどは日本語で書く
- ソースコードには適度にコメントを書く
  - コメントは以下の内容に限る
    - 「なぜそうするか (設計上の制約、トレードオフ、バグ回避理由)」
    - 「何をしているか (数行以上のコードの塊に対して短い要約として)」
- ドキュメントは、コードを見ただけではわからない情報や、コードの背後にある意図や設計の理由などを書く
  - ソースコードなどを見ればすぐわかることは書かない

## テスト方針

簡潔さ・安定性・速度を重視する。

- 簡潔さ
  - テストダブル (モック・スタブ・モンキーパッチなど) は可能な限り避けて実コードを通す
  - 単純過ぎるテストは避け、ある程度まとまった機能のテストを重視する
    - private関数などを直接テストすることは極力避ける (ただしこれを重視しすぎて設計をゆがめるのは厳禁)
  - 見通しをよくするため、テスト関数は1つのテスト対象に対して原則1つにまとめる
    - 可能な場合はCRUDなどの一連の流れをまとめて1つのテスト関数にする
  - テスト間の影響を避けるため、テストデータは可能な限り後始末する
  - 本質的でない処理 (テストデータのセットアップなど) は可能な限り共通化する
- 安定性: flakyテスト防止
  - 現在時刻・乱数・UUIDなど非決定的な値は、引数やDIで外部から注入可能な設計にする。テストでは固定値を渡す
  - sleepはタイミング依存でflakyになりやすいため極力避ける。イベント・コールバック・ポーリングなど確定的な待機手段を優先する
  - ファイルシステム・ネットワーク・外部サービスへの依存は、可能な限りテスト用の一時ディレクトリやインメモリ実装で代替する
  - 並行実行を前提に、ポート番号・ファイルパスなどの競合が起きない設計にする
  - 環境依存 (OS、ロケール、タイムゾーンなど) で結果が変わるテストは、テスト内で明示的に条件を固定する
- 速度
  - 全体の実行時間の短さを常に意識する
  - コストの高いセットアップ (DB初期化、大きなファイル生成など) はテストスイート内で1回だけ実行し、複数テストで共有する
    - ただしテスト間の状態依存や実行順序依存を持ち込まないよう注意する
  - テストデータは機能を検証できる最小限にとどめる (巨大ファイルや大量レコードは避ける)
  - やむを得ずsleepを使用する場合、0.1秒単位で安定する最小限にし、関数全体でも1秒以内を目標とする

## ファイル構成

- `.claude/rules/agent.md` は汎用的なエージェント向けベース指示であり、ユーザーから明確な指示がない限り、このファイルを変更してはならない
- プロジェクト固有の指示は `CLAUDE.md` に書く
